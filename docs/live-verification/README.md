# REL8TION Live Verification

This folder contains a safe, read-only verification system for comparing current repo assumptions against live Supabase and the currently configured Vercel app routes.

It is intentionally conservative:

- It does not write to Supabase.
- It does not modify production data.
- It does not call SMS, outreach, sync, or mutation-style Edge Functions.
- It does not print full keys or tokens.
- It marks unknown items as `NEEDS_VERIFICATION` unless a read-only check can prove the item is missing.

## Files

- `expected-supabase-contract.json` is the repo-local contract the verifier checks against.
- `verify-supabase-live.mjs` runs the read-only checks and writes reports.
- `live-verification-checklist.md` is the manual checklist for schema, RLS, functions, and product assumptions.
- `verify-vercel-routes.md` is the route checklist for the current Vercel/static app surface.
- `latest-live-verification-report.json` and `latest-live-verification-report.md` are generated by the script and should be treated as point-in-time reports.

## Status Labels

- `PASS` means the read-only check succeeded under the available access.
- `WARN` means the check found partial evidence or a condition that needs review.
- `NEEDS_VERIFICATION` means the script cannot prove the item from safe anon/local checks.
- `FAIL` means the script has evidence that an expected file/table/column is missing or unreadable in a way that should be investigated.

Important: `PASS` does not prove that deployed code, RLS policies, real writes, or current Supabase production data are healthy. It only proves the specific read-only check that ran.

## How To Run

From the repo root:

```powershell
$env:SUPABASE_URL="https://YOUR-PROJECT.supabase.co"
$env:SUPABASE_ANON_KEY="YOUR_ANON_KEY"
npm run verify:live
```

Optional privileged schema confirmation:

```powershell
$env:SUPABASE_SERVICE_ROLE_KEY="YOUR_SERVICE_ROLE_KEY"
npm run verify:live
```

The optional service-role key is used only for zero-row read probes. The script still does not write, update, delete, call SMS functions, or call outreach functions.

## Outputs

The script writes:

- `docs/live-verification/latest-live-verification-report.json`
- `docs/live-verification/latest-live-verification-report.md`

Do not treat live Supabase as verified until the script has been run with real environment variables and the generated report has been reviewed.

## What This Cannot Prove

- Full RLS policy semantics.
- Edge Function deployment state.
- Vercel production deployment state.
- Real buyer check-in, SMS, outreach, or NFC behavior.
- Data quality in `listing_agents`, `open_houses`, or outreach queues.

Those require dashboard inspection, deployment inspection, and separate non-destructive operational checks.
